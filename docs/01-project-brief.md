# ClawForge 项目定位

项目暂名：ClawForge 社区开荒者  
参赛方向：多 Agent AI 数字团队  
一句话：一支面向早期 Agent 社区的自动开荒数字团队，能发现需求、制造技能、自动发布、图文宣传、播种悬赏、连接 Agent，并把社区反馈继续炼成下一轮增长。

## 1. 核心洞察

EasyClaw 是一个 Agent 技能共享网络，但目前仍处在早期社区阶段：

- 论坛有内容，但需求信号稀疏。
- 技能市场有 698 个公开技能，但质量、重复度、调用分布差异很大。
- 赏金任务能形成业务闭环，但当前 open 任务较少。
- 平台 API 和 A2A 协议已经比较完整，只是缺一个把这些能力串起来的“生态增长引擎”。

所以 ClawForge 不做“又一个助手”，也不只是做“技能工厂”，而做一套 Agent 社区开荒基础设施：

> 把社区里微弱的需求信号放大，把一次性经验沉淀成可复用技能，把技能发布成社区事件，把使用反馈再变成下一轮内容、悬赏、合作和技能进化。

## 2. 面向评委的主张

传统 Agent 作品通常展示“我能完成一个任务”。  
ClawForge 展示的是“我能让 Agent 社区自己开荒、建设和增长”。

这和比赛主题高度契合：

- 有明确的 AI 数字团队。
- 有真实平台业务场景。
- 有完整 Pipeline，从需求发现到技能发布，再到论坛传播和反馈进化。
- 有技术深度：数据抓取、需求聚类、Agent 编排、Skill 生成、自动测试、反馈闭环。
- 有产品眼光：解决早期社区冷启动，而不是只做炫技 Demo。
- 有平台价值：直接帮助 EasyClaw 增加高质量技能、帖子、悬赏、教程和 Agent 之间的连接。

## 3. 项目边界

MVP 目标：

- 做一个可演示的 Web 控制台。
- 接入 EasyClaw 公开 API。
- 用多 Agent 流程从信号中生成一个 Skill 候选。
- 自动给候选 Skill 打分、测试、生成发布材料。
- 支持“草稿模式”和“发布模式”两种路径：草稿模式只导出，发布模式在配置 API Key 后可发技能、发论坛宣传帖。
- 用可视化方式展示整个流水线。

不在 MVP 中强依赖：

- 高频定时任务。
- 大规模生产级爬虫。
- 真实支付、积分、私信等敏感操作。

可选增强：

- 登录后使用 API Key 进行 skill.publish、forum.post、forum.comment、bounty.create。
- 接入外部网页作为需求源。
- 自动生成演示视频或截图包。
- 生成技能封面图、宣传长图、论坛图文帖。
- 给优秀 Agent 发送合作邀请草稿。

## 4. 目标用户

主要用户：

- EasyClaw 新 Agent 主人：不知道该做什么技能、怎么写技能。
- 平台运营者：想知道社区缺什么、哪些技能值得扶持。
- 技能作者：想把一次经验沉淀成更规范的 Skill。

比赛评委视角：

- 能看到你对平台真实生态的理解。
- 能看到多 Agent 协作不仅是角色命名，而是有输入、输出、验收和迭代。
- 能看到项目可以继续长成一个真正的产品。

## 5. 展示口号

首页主标语：

> 让早期 Agent 社区自动开荒。

副标题：

> ClawForge 从论坛、技能市场、赏金任务和 API 文档中发现稀疏需求，由多 Agent 团队生成技能、自动验收、发布推广、播种讨论，并把反馈炼成下一轮增长。

技术宣言：

> The product is not a skill. It is a community-building agent team.

## 6. 成功标准

演示成功标准：

- 评委能在 30 秒内明白项目解决什么问题。
- 评委能看到至少 5 个 Agent 的协作过程。
- 评委能看到真实 EasyClaw 数据被读入。
- 评委能看到从需求到 Skill 的完整产物。
- 评委能看到质量门禁，而不是纯文本生成。
- 评委能看到下一轮进化依据。
- 评委能看到 Skill 发布后如何变成社区内容、悬赏和协作网络。

项目成功标准：

- 能跑通一个完整案例。
- 能保存每次运行的报告。
- 能导出 Skill Markdown、测试报告、发布文案。
- UI 有足够强的“比赛表现力”。
